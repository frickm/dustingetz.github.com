"confident code" vs "timid code" 

 

Avdi Grimm articulates several metaphors for thinking about well-constructed code at the tactical level. This is in essense a discussion of essential vs accidental complexity, but without sounding so academic. Thanks Avdi, talking about software quality in terms of complexity seems to only turn out well when the participants already understand accidental complexity. Looking forward to using the metaphors Avdi established. 

 

Avdi defines "narrative structure", as the opposite of "haphazard structure". Well written code tends to read as if it was narrated to a future maintainer. "Gather input, Perform work, Deliver results, Handle errors. In that order!"

My notes quoted or paraphrased from his slides below. 

 

Timid Code: 

- Lives in fear  

- Constantly second-guessing itself  

- Randomly mixes input gathering, error handling, and business logic  

- Imposes cognitive load on the reader  

 

"Nil checks are the most common form of timid code" 

 

tricks to write confident code: 

- list-ify -- high level data structures lead to high level control flow 

- asserts and preconditions. asserts as documentation. 

- coerce liberally [disagree mostly, i'd prefer to make my value types well-defined at all times] 

- dynamically extend objects [like a monkey patch? that's a stretch..] 

- iterative style [i've been calling this functional style, or data-oriented style] 

- narrative code seperates business logic from input gathering. your business logic thus has only business conditionals!
