java sucks, because it makes functional idioms frustratingly heavy to use 

 

TLDR: certain abstractions are so heavy in java, that most java programmers don't bother -- they just inline some for loops and live with the increased complexity [Odersky 1]. 

 

idiomatic java leads to accidental complexity 

 

Lets check if a string contains uppercase characters, in scala and in java. 

 

// *djg* these two snippets are from [1] 

String str; 

 

// idiomatic scala -- functional style 

val containsUpperCase = str.exists(_.isUpper); 

 

// idiomatic java -- imperative style 

boolean containsUpperCase = false; 

for (int i = 0; i < str.length(); ++i) { 

if (Character.isUpperCase(str.charAt(i))) { 

containsUpperCase = false; 

break; 

} 

}  

 

It's self-evident that the functional idiom is going to have less defects, because the functional idiom is easier to quickly understand. But this code isn't a problem in Java, this is just the advantage of using functional idioms. Java's standard libraries don't ship with APIs for idiomatic functional collection algorithms, but of course idiomatic functional control abstractions are possible. Java should have this stuff in their library, but as a library defect and not a language defect I think it's a superficial complaint. 

 

you can write java in functional style, but for many programmers its too heavy or annoying to bother 

 

Lets implement Collection.exists ourself or use a third-party library, and write some functional-style java: 

 

// java using functional style 

boolean containsUpperCase = CollectionUtil.exists(str,  

Character.isUpperCase); // can't do this :( java doesn't have first class functions :( :( 

new Predicate<Character>() { 

public boolean apply(Character ch) { 

return Character.isUpperCase(ch); 

} 

}); 

 

it's just... heavier. Yes, in verbosity, but that's superficial: i think good programmers spend way more time thinking, than typing[2]. Its heavier in terms of abstractions you have to think about -- you have to look up and implement the Predicate interface and write out all the boilerplate and line up all your types. There's just cognitive overhead when our mental solution of the problem is simply `str.exists(_.isUpper)`. Oh, wait, that's also the solution in scala! hmmmm.. 

 

in java, nontrivial functional idioms have absurd boilerplate that looks like line noise 

 

Lets take a more complicated example that's in production at work: 

 

public Collection<?> getData(final IHttpRequest request, OrmContext ormcontext) throws Exception 

{ 

CollectionUtil.ListMutator_2<IDictionaryEntry, Map<String, Object>> toValueAsNameOptions =  

new CollectionUtil.ListMutator_2<IDictionaryEntry, Map<String, Object>>() { 

public Map<String, Object> apply(IDictionaryEntry entry) { 

 

// if we want a dict to post as a string (disregarding the real value, e.g. for  

// dctm string attrs whose picklist is defined in a dictionary), 

// mutate from IDictionaryEntry to Map<String, Object> so we can override the value field 

 

Map<String, Object> selectOptions = new HashMap<String,Object>(); 

selectOptions.put("name", entry.getLabel(request.getLocale())); 

selectOptions.put("value", entry.getLabel(request.getLocale())); 

return selectOptions; 

} 

}; 

 

@SuppressWarnings("unchecked") // cast keeps Collection<?> happy 

Collection<IDictionaryEntry> dictEntries = (Collection<IDictionaryEntry>) super.getData(request, ormcontext); 

 

return CollectionUtil.transform(dictEntries, toValueAsNameOptions); 

} 

 

Lol. This takes actual mental effort to get all the types lined up. ListMutator maps an input element to an output of the same type, ListMutator_2 maps input to output of a different type. And there's a random unchecked cast, that I couldn't resolve in a 20 minute deeper look. Java, I don't want to waste my time thinking about casts and boilerplate, i want to solve my goddamn problem, get the solution right on the first try, and move on! 

 

Despite Java's suckage, there's huge value in writing java with functional style 

 

The business logic is a one liner. I can look at this code for 30 seconds and know exactly what it is supposed to do, and know that the approach is fine, without caring about the implementation of toValueAsNameOptions. The implementation is totally seperate from the business logic, and I can send a code review to my super-busy colleagues and they can confirm it's right in 30 seconds without brain-parsing a bunch of for loops. 

 

the code abstractions exactly line up with how i would explain what needs to happen to my customer: "dictionaries usually render to combobox datastores as name/value pairs where the value is an ID code, but here we want the combo to think of his value as a raw string, so transform the data". 

 

I don't want to beat a dead horse, so I'll link without discussion another heavily functional example of generating non-trivial SQL for a datstore, which I think is well-written code. 

 

in java, it's easier to bang out a new loop than it is to inspect state with predicates (or, what the fuck does this code do?) 

 

When we're persisting business state, we need to check business invariants. Here, we have dynamically configurable wizards, and if you are trying to change wizard metadata, we need to verify that after the change, there is at most one "active" wizard per each "level". This was sent to me in a code review from my "uber tech lead" who is one of the best software engineers I've ever met. (Don't stare too long, just let your eyes glaze over and scroll to my next paragraph) 

 

@Override 

protected void doSave(IWizard wizard, OrmContext ormcontext) throws Exception  

{ 

// If we are trying to save an active wizard, then make sure we're the only one 

if (wizard.getIsActive().getBooleanValue()) 

{ 

Level level = wizard.getLevel(); 

DebugUtil.verify(null != level); 

 

// Get active wizards for this level 

List<IWizard> activeWizards = getActiveWizardsForLevel(ormcontext, level); 

 

// If this object is not new, then it might already be on that list -- remove it so we don't 

// block ourselves from modifying *the* active wizard 

if (!Wizard.isNew()) 

{ 

for (IWizard checkWiz : activeWizards) 

{ 

if (checkWiz.getId().equals(wizard.getId())) 

{ 

activeWizards.remove(checkWiz); 

break; 

} 

} 

} 

 

// There are still active wizards for this level :( 

if (!activeWizards.isEmpty()) 

{ 

// handle error 

} 

} 

 

super.doSave(wizard, request, appContext); 

} 

 

So I looked at the code review and was like, hell, I have no idea if this is right and can't possibly approve this change without stepping through this in my head.  

 

all we want is to be able to quickly say, "yep, this code is correct" 

 

Here's what he's accomplishing in if you think about these things without using mutable data and side effects: 

 

Predicate<IOrmObject> isSameObjectAsWizard = new Predicate<IOrmObject>() { 

public boolean apply(IOrmObject o) { 

return o.getId() == wizard.getId(); 

} 

}); 

 

if (activeWizards.isEmpty() 

|| (activeWizards.size() == 1 && CollectionUtil.exists(activeWizards, isSameObjectAsWizard))) // [3] 

{ 

super.doSave(wizard, request, appContext); 

} 

else 

{ 

// handle error 

} 

 

If you got that in a code review, you look at it, you say "yep he understood the business constraint, this is correct" and you are done. Way less likely for there to be a defect. Even with the friction of java predicates, it's still better code because it's easier to verify correct. 

 

java created a whole industry of programmers who have exactly one hammer in their toolkit 

 

Java sucks because there is so much friction in doing anything but making a class, so when faced with a problem, someone trained only java will think, "Oh! I'll make a new class!". Steve Yeggie nailed it in his famous rant, Execution in the Kingdom of Nouns: "The residents of the Kingdom of Java aren't merely happy â€” they're bursting with pride!"[4] 

 

Java sucks because Java makes it so hard to write elegant code, that most programmers don't even realize that there is a better way. That really is Java's fault -- not the developer's -- a developer's job is to bang out features for the product owner to sell, not to masturbate over how to write the most elegant 10-liner. When your boss is peeking in over your shoulder asking when it will be done, he doesn't want to hear that you're spending a day to consolidate your thoughts on reducing defect long-term defect rates on this project, he wants to know if we're going to deliver on his absurdly aggressive short-term schedule. The catch is, compounding technical debt isn't transparent and measurable, and before you know it, you no longer hanve enough control to execute quickly, or as high quality, or our margins are lower. Even though, in the long term, I do speculate that a team in control of complexity will consistently out-perform a team who is struggling to keep it in check.
